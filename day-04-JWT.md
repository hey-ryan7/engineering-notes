# Day 4 — JWT (JSON Web Token)

## Why JWT Exists

JWT was created to solve some limitations of session-based authentication. In traditional session-based systems, the server must store and manage session data for every logged-in user. As applications grow and handle more users, managing sessions across multiple servers becomes more complex.

JWT provides a stateless authentication approach, allowing the server to verify a token without storing session data for every user.

---

## What is JWT

JWT (JSON Web Token) is a signed token that contains information about a user or an authentication claim. The server can verify the token's signature to ensure that the token has not been modified.

If the signature is valid, the server can trust the information stored inside the token.

---

## Why Is It Called a Token?

A token is something that grants access to a resource or proves a specific claim.

A JWT acts as a token because it allows a user to prove their identity and access protected resources when the token is valid.

---

## JWT Structure

A JWT consists of three parts:

```text
Header.Payload.Signature
```

---

### Header

The header contains metadata about the token, such as:

* Token type (JWT)
* Signing algorithm

Example:

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

---

### Payload

The payload contains claims or information about the user.

Common examples:

* User ID
* Email
* Username
* Expiration time

Sensitive information such as passwords should never be stored in the payload.

The payload is Base64URL encoded, not encrypted, which means anyone can decode and read it.

---

### Signature

The signature is generated by the server using a secret key and the token data.

The server verifies the signature to ensure that the token has not been tampered with.

If the signature is invalid, the token is rejected.

---

## How JWT Works

1. The user submits login credentials.
2. The server validates the credentials.
3. If the credentials are correct, the server generates a JWT.
4. The server sends the JWT back to the client.
5. The client stores the token.
6. When accessing protected resources, the client sends the token in the Authorization header.
7. The server verifies the token's signature.
8. If the token is valid, the server processes the request and returns the requested data.

---

## JWT vs Session

### JWT

* Stores user information inside the token.
* Usually does not require server-side session storage.
* Scales well across multiple servers.
* Uses signatures to verify authenticity.

### Session

* Stores session data on the server.
* The browser usually stores only the Session ID.
* Requires server-side storage such as memory, Redis, or a database.
* The server must look up the session for each request.

---

## Advantages of JWT

* Stateless authentication
* Easier to scale across multiple servers
* No server-side session lookup required
* Useful for distributed systems and APIs

---

## Disadvantages of JWT

* Revoking tokens can be difficult before expiration
* Compromised tokens remain valid until they expire
* Token management becomes more complex when implementing logout, rotation, and revocation strategies
